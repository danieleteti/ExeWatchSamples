External Service Calls

Real applications depend on external services. These simulations show how ExeWatch tracks each call with timing, success/failure counters, and error logging — so you know exactly which service is causing problems.

Payment Gateway

40% fail

Simulates a payment of EUR 149.99. Can succeed (60%), timeout after 4s (20%), be declined (15%), or return a 502 error (5%).

service.payment_gateway payment.success payment.timeout

Email Service

25% fail

Sends an order confirmation email. Can be rate-limited (429) or fail with an SMTP connection error after 2s timeout.

service.email email.sent email.rate_limited

Geocoding API

~85% OK

Geocodes an address. Has a 15% cache hit (very fast), normal responses, and occasional quota-exceeded errors.

service.geocoding geocoding.cache_hit geocoding.quota_exceeded

Service Orchestration

Calls 5 services sequentially inside a parent timing. Each service is timed individually, and the parent timing captures the total duration. Failures are counted and logged — exactly how a real microservice orchestrator works.

What to look for in ExeWatch:
  • Timing page — Compare response times across services. Identify which ones are slowest.
  • Metrics — Counter metrics like "payment.success", "payment.timeout", "email.rate_limited" show trends over time.
  • Logs page — Filter by tag "payment", "email", or "geocoding" to see per-service activity.
